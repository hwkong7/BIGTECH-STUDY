# ER 다이어그램(ER diagram)

<aside>
💡

데이터베이스를 구성하는 요소들의 관계 즉, 엔티티 관계(Entity Relationship(ER))를 나타내는 그림

</aside>

## 목적

- 저장되는 **엔티티의 구조를 모델링**하는 것
  - 데이터베이스로 표현할 대상을 시각적으로 설계
  - ⇒ 추후 데이터베이스를 확장/수정할 때 영향 범위를 쉽게 파악 가능
  - ⇒ 유지보수 용이, 개발자 간 소통 원활
  - 반대로, **모델링 없이 만들면**
    - = 첫 단추를 잘못 꿰는 것
    - 언젠가는 많은 시간과 노력 ⇒ 구조 수정
    - 비효율적인 설계는 데이터 중복, 불일치 가능성을 높임
    - => **질의 성능도 저하**될 수 있음

## 식별/비식별 관계

### 식별 관계

- 참조되는 엔티티가 존재해야만 참조하는 엔티티도 존재할 수 있는 관계
- 참조되는 테이블의 기본 키를, 참조하는 테이블의 **외래 키이자 기본 키**로 사용하는 경우

### 비식별 관계

- 참조되는 엔티티가 없어도 참조하는 엔티티가 존재할 수 있는 관계
- 참조되는 테이블의 기본 키를, 참조하는 테이블의 **일반 외래 키**로만 사용하는 경우

## 제2 정규형(2NF)

#### 필요충분조건

- **제1 정규형을 만족**하고
- **기본 키가 아닌 모든 필드가 기본 키 전체에 완전히 종속**되어야 함
- 기본 키가 2개 이상의 필드로 구성될 때 고려
- 기본 키의 일부에만 종속되는 필드가 있으면 제거/분리해야 함
- => 모든 필드가 기본 키 전체에 종속되어야 함

#### 부분 함수 종속성

- 기본 키가 아닌 필드가 기본 키의 **일부에만** 종속되어 있는 경우

#### 완전 함수 종속성

- 기본 키가 아닌 필드가 기본 키 **전체에** 종속된 경우

#### 부분 함수 종속성이 없는 상태 = 완전 함수 종속인 상태

- 후보 키에 속하지 않는 모든 필드가 기본 키에 완전 함수 종속인 상태

![image.png](attachment:d3abb02a-4297-4e12-bf16-b56e79acd437:image.png)

- ‘가격’은 ‘구매 항목’과의 종속 관계가 있을 뿐, ‘회원 ID’와는 아무런 관계가 X
- 일부 기본 키에만 종속되어 있고, 기본 키 일부에만 종속된 필드가 존재하므로 제2정규형을 만족X

![image.png](attachment:15164205-52eb-4e23-9f04-24d5a46ef309:image.png)

  - 만족하려면 기본 키에 완전히 종속되는 필드로만 테이블을 구성해야 하므로 다음과 같이 테이블을 분할

## 제3정규형

<aside>
💡

- **제2 정규형을 만족**하면서
- 기본 키가 아닌 모든 필드가 기본 키에 **이행적 종속성**이 없어야 함
</aside>

#### 이행적 종속 관계(transitive dependency)

- 어떤 테이블에 A, B, C가 있을 때
- A→B, B→C 이면 A→C가 성립
- 이때 A와 C 사이에는 이행적 종속성이 있다고 함
- A→B 이고, B→C일 때 A→C를 만족하는 관계

![‘학과’와 ‘학과 사무실 위치’에 대한 정보가 ‘학번’과 함께 저장되어 잇는 테이블](attachment:f2197e0a-849e-41bb-9afb-435d2eed03f5:image.png)

‘학과’와 ‘학과 사무실 위치’에 대한 정보가 ‘학번’과 함께 저장되어 잇는 테이블

- 예)
  - `학번 → 학과`
  - `학과 → 학과 사무실 위치`
  - 그러면 `학번 → 학과 사무실 위치`
  - => `학과 사무실 위치`는 기본 키에 직접 종속된 게 아니라 간접 종속
  - => 3NF 위반
- 즉,
  - 기본 키가 아닌 나머지 필드들은 간접적으로도 종속되면 안 됨
  - 기본 키가 아닌 필드들끼리 서로를 결정할 수 없어야 함

![제3 정규형 만족하는 테이블](attachment:d2bb7784-a2dd-4a3b-bb0f-b222f96bc844:image.png)

제3 정규형 만족하는 테이블

해결

- `학번-학과` 테이블
- `학과-학과 사무실 위치` 테이블
- => 이렇게 분리

## 보이스/코드 정규형(BCNF(Boyce-Codd Normal Form)

<aside>
💡

- **제3 정규형을 만족**하면서
- **모든 결정자가 후보 키여야 하는 정규형**
</aside>

### 결정자

- 특정 필드를 식별할 수 있는 필드
- A가 B를 결정하면(A→B), A는 B의 결정자

![image.png](attachment:4e764afd-10dd-4ad2-baa2-3ff0f6d4d831:image.png)

![image.png](attachment:e895ddc9-87a2-4472-8951-d29ce78ccc5d:image.png)

예)

- 이행적 종속 관계는 없어서 3NF는 만족
- 하지만 `담당 교수`가 `과목 코드`를 결정하는데 `담당 교수`는 후보 키가 아님
- => BCNF는 만족하지 않음

해결

- `학번-과목 코드` 테이블
- `과목 코드-담당 교수` 테이블
- => 분리 필요

## 제4정규형

- A↠B가 존재할 경우, R의 모든 속성이 A에 함수적 종속 관계를 만족하는 정규형

### 다치 종속 제거

- 다중 값 종속(MVD: Multi Valued Dependency)
- A, B, C 속성을 가진 테이블 R에서
- (A, C)에 대응하는 B값의 집합이 A에만 종속되고 C에는 무관하면
- B는 A에 다중 값 종속
- => `A↠B`로 표기

## 제5정규형

- 테이블 R의 모든 조인 종속이 R의 후보 키를 통해서만 성립되는 정규형

### 조인 종속(JD: Join Dependency)

- 어떤 테이블 R의 속성 부분집합 X, Y, ..., Z가 있을 때
- 테이블 R이 자신의 프로젝션 X, Y, ..., Z를 모두 조인한 결과와 동일하면
- R은 조인 종속 JD(X, Y, ..., Z)를 만족한다고 함

## 장단점

- 정규화를 거듭할수록 데이터는 깔끔하게 정돈됨
- 이상 현상도 줄어듦
- 하지만 정규화를 너무 많이 하면 오히려 성능 저하 가능

# 역정규화(denormalization)

<aside>
💡

- 검색 속도를 높이기 위해 분할된 테이블을 다시 하나로 합치는 작업
- 경우에 따라 정규화를 하지 않거나, 정규화된 테이블을 역정규화하기도 함
</aside>

- 정규화를 거듭하면 테이블이 많이 쪼개질 수 있음
  - 테이블 多 ⇒ 조인 연산이 빈번해지고, 다른 테이블을 참조하기 위한 성능상의 비용 ↑
- NoSQL에서는 기본적으로 정규화X
