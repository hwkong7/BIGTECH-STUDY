# RDBMS vs NoSQL: NoSQL의 특징

## NoSQL(Not Only SQL)

- RDBMS처럼 테이블 형태 + SQL만 사용하는 방식과 달리
- 레코드를 다양한 형태로 저장할 수 있고
- SQL 이외의 방식으로 저장된 데이터도 다룰 수 있음

## 키-값 데이터베이스

<aside>
💡

- 이름 그대로 데이터베이스에 레코드를 키(필드)와 값의 쌍으로 저장하는 데이터베이스
- 가장 간단한 형태의 NoSQL 데이터베이스 유형 - 예) Redis, Memcached
</aside>

- 구조가 단순해서
  - 보조기억장치가 아니라 **메모리**에 저장하는 경우도 많음
  - => **인메모리 데이터베이스(In-memory database)**
- 주로 캐시, 세션처럼 가벼운 정보 저장에 많이 사용

## 도큐먼트 데이터베이스(document database)

= 문서지향 데이터베이스(document-orieented)

### 도큐먼트

- ‘정형화되어 있지 않은 NoSQL 레코드의 단위’를 통칭하기 위해 사용하는 용어
- 많은 데이터베이스에서 JSON이나 XML과 같은 형식을 도큐먼트로 활용한다.
  - 예) MongoDB

### 컬렉션(collection)

- MongoDB에서는 도큐먼트가 모인 것 → 데이터베이스

![image.png](attachment:998c8aed-1f79-4d23-8077-cfdfbafba1c5:image.png)

- MongoDB - JSON 형태
  - 키가 필드 JSON 데이터 하나 하나가 RDBMS의 행

#### 차이점

- RDBMS와는 달리 고정된 스키마가 없음 ⇒ 각 도큐먼트가 유연한 스키마를 가질 수 있음

## 그래프 데이터베이스(graph database)

<aside>
💡

데이터베이스에 저장하고자 하는 데이터를 그래프의 노드 형태로 저장하는 데이터베이스

- 예) neo4j
</aside>

## 칼럼 패밀리 데이터베이스(column family database)

<aside>
💡

관련된 컬럼들의 집합을 하나의 단위로 묶어서 관리하는 방식

- 예) Cassandra, HBase
</aside>

- RDBMS와 같이 행(row)과 열(column)이라는 개념이 있고, RDBMS에서 키를 통해 특정 행을 식별하듯 로우 키(row key)를 통해 특정 행을 식별

![image.png](attachment:3d74e307-9147-48da-af76-ceb52b6aae79:image.png)

- 다만, RDBMS와는 다르게 **정규화나 조인을 사용하지 않고**, **스키마가 고정되어 있지 않아** 자유롭게 열을 추가 가능
- ⇒ 동적으로 변하는 열 구조에 더 가까움

#### **칼럼 패밀리(column family)**

<aside>
💡

관련 있는 열들의 묶음

</aside>

#### **키스페이스(keyspace)**

<aside>
💡

- 여러 칼럼 패밀리를 포함하는 최상위 단위
- 일반적으로 애플리케이션마다 하나씩 사용
</aside>

- 키스페이스 안에 여러 칼럼 패밀리가 있고
- 칼럼 패밀리 안에는 여러 열이 포함될 수 있음
- 이 열들은 자유롭게 추가/삭제될 수 있음
- => RDBMS 테이블보다 더 자유로운 형태로 레코드 관리 가능

## NoSQL의 주요 이점

- 높은 부하 처리에 유리
- 대용량 데이터를 다루는 분산 환경에서 강점이 큼
- 핵심 장점
  - 확장성
  - 유연성
  - 가용성
  - 성능

#### **수평적 확장(Scale-out)**

- 대부분의 NoSQL은 **스케일 아웃**을 주요 목표로 설계됨
- => 서버 여러 대로 확장하기 쉬운 경우가 많음
- => RDBMS보다 수평 확장에 유리한 편

#### RDBMS vs NoSQL

| 구분            | RDBMS                                           | NoSQL                                                        |
| --------------- | ----------------------------------------------- | ------------------------------------------------------------ |
| 데이터 구조     | 정형 데이터에 적합                              | 비정형 또는 반정형 데이터에 적합                             |
| 스키마          | 고정된 스키마 사용                              | 유연한 스키마 사용                                           |
| 무결성 / 일관성 | 무결성과 일관성 보장이 강함                     | 상대적으로 약할 수 있음                                      |
| 트랜잭션        | ACID를 엄격하게 준수                            | ACID를 엄격히 지키지 않는 경우가 많음                        |
| 성능            | 복잡한 관계 처리에 유리                         | 대용량 처리와 빠른 성능에 유리                               |
| 확장성          | 수직 확장에 주로 적합                           | 수평 확장에 유리                                             |
| 관계 표현       | 테이블 간 관계 표현에 강함                      | 관계형 데이터 표현에는 상대적으로 약함                       |
| 적합한 경우     | 금융, 결제, 재고 관리 등 정확성이 중요한 경우   | SNS, 로그 데이터, 실시간 분석 등 대용량/유연성이 중요한 경우 |
| 선택 기준       | 정형화된 데이터, 무결성·일관성이 중요할 때 선택 | 비정형 데이터, 성능·확장성이 중요할 때 선택                  |
| 대체 가능성     | NoSQL로 완전히 대체되기 어려움                  | RDBMS를 완전히 대체하기 어려움                               |

- 데이터의 **정확성, 일관성**이 중요하면 -> **RDBMS**
- **성능, 확장성**이 중요하면 -> **NoSQL**

# 다양한 NoSQL: MongoDB와 Redis 맛보기

## MongoDB

- 레코드는 도큐먼트 단위로 저장 → 레코드가 모여 컬렉션 → **컬렉션**이 모여 → **데이터베이스**

### 데이터베이스 생성/사용 & 컬렉션 생성

```sql
use 데이터베이스_이름                # 데이터베이스 생성/사용
db.createCollection("컬렉션_이름")   # 컬렉션 생성
```

#### 특정 컬렉션의 도큐먼트를 확인하는 명령

```sql
db.컬렉션_이름.find()
```

| 연산자  | 설명                              | 코드                               |
| ------- | --------------------------------- | ---------------------------------- |
| $eq     | 지정한 값과 같은 값 찾기          | { "field": { "$eq": value } }      |
| $ne     | 지정한 값과 같지 않은 값 찾기     | { "field": { "$ne": value } }      |
| $gt     | 지정한 값보다 큰 값 찾기          | { "field": { "$gt": value } }      |
| $gte    | 지정한 값보다 크거나 같은 값 찾기 | { "field": { "$gte": value } }     |
| $lt     | 지정한 값보다 작은 값 찾기        | { "field": { "$lt": value } }      |
| $lte    | 지정한 값보다 작거나 같은 값 찾기 | { "field": { "$lte": value } }     |
| $and    | 모든 조건이 참인 문서 찾기        | { "$and": [ {조건1}, {조건2} ] }   |
| $or     | 하나라도 참인 조건의 문서 찾기    | { "$or": [ {조건1}, {조건2} ] }    |
| $not    | 조건이 거짓인 문서 찾기           | { "field": { "$not": condition } } |
| $nor    | 모든 조건이 거짓인 문서 찾기      | { "$nor": [ {조건1}, {조건2} ] }   |
| $exists | 필드 존재 여부 확인               | { "field": { "$exists": true } }   |
| $type   | 특정 데이터 타입 확인             | { "field": { "$type": type } }     |
| $sum    | 합계 계산하기                     | { "$sum": "$field" }               |
| $avg    | 평균 계산하기                     | { "$avg": "$field" }               |
| $min    | 최소값 계산하기                   | { "$min": "$field" }               |
| $max    | 최대값 계산하기                   | { "$max": "$field" }               |

## Redis

- 키-값 데이터베이스이자, 인 메모리 데이터베이스의 일종
  - 문자열과 리스트, 해시 테이블, 집합 등

![image.png](attachment:237a9ea8-e239-4cbf-b050-0d068f8719d1:image.png)

| 종류  | 설명                                        |
| ----- | ------------------------------------------- |
| SET   | 문자열 값 저장                              |
| SETNX | 키가 존재하지 않는 경우에만 문자열 값 저장  |
| GET   | 문자열 값 조회 (키를 통해 대응되는 값 조회) |
| MGET  | 여러 문자열 값 조회                         |
| DEL   | 키 삭제                                     |

| 종류   | 설명                                        |
| ------ | ------------------------------------------- |
| LPUSH  | 리스트의 왼쪽(앞)에 새로운 요소 추가        |
| RPUSH  | 리스트의 오른쪽(뒤)에 새로운 요소 추가      |
| LPOP   | 리스트의 왼쪽(앞)에서 요소를 제거 후 반환   |
| RPOP   | 리스트의 오른쪽(뒤)에서 요소를 제거 후 반환 |
| LRANGE | 지정된 범위의 요소들을 반환                 |
| LLEN   | 리스트 길이 반환                            |
