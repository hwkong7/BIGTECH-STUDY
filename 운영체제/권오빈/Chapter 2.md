# 2장 프로세스와 스레드

<aside>

프로세스의 유형에는 사용자가 보는 공간에서 사용자와 상호작용하며 실행되는 **포그라운드 프로세스(foreground process)**와 백그라운드 프로세스**(background process)**가 있다.

백그라운드 프로세스 중 사용자와 별다른 상호작용 없이 주어진 작업만 수행하는 특별한 백그라운드 프로세스를 **데몬(daemon)**이라고 한다.

![image.png](attachment:20e638e4-2704-4290-9c00-37a0caa9aeb0:image.png)

**코드 영역(code segment)**은 실행될 프로그램 코드가 적재되는 영역이다.
**데이터 영역(data segment)**은 프로그램이 실행되는 동안 유지할 데이터가 저장되는 공간이다. 정적 변수, 전역 변수가 저장된다.
**힙 영역(heap segment)**은 프로세스의 실행 도중 동적으로 사용할 수 있도록 할당된 공간이다.
**스택 영역(stack segment)**은 일시적으로 사용할 값들이 저장되는 공간이다. 함수의 실행이 끝나면 사라지는 매개변수,. 지역 변수, 함수 복귀 주소 등이 저장된다.

</aside>

## PCB와 컨텍스트 스위칭

<aside>

**프로세스 제어 블록(Process Control Block, PCB)**는 프로세스에 관한 정보를 저장하는 구조체이며 프로세스 당 하나씩 존재한다.
프로세스가 생성될 때 만들어지고, 프로세스가 종료되면 삭제된다. 
프로세스 테이블에 의해 관리된다.

![image.png](attachment:0f420928-4f5a-4063-af4d-7bfe4a369662:image.png)

**컨텍스트(context)**란 한 프로그램이 실행 중인 일체의 상황 혹은 상황 정보이다.
CPU가 현재 프로그램의 실행을 중지하고 다른 프로그램을 실행할 때 현재 실행중인 프로그램의 컨텍스트를 메모리에 저장하고 새로 실행시킬 프로그램의 저장된 컨텍스트를 CPU에 복귀시킨다.
이를 **컨텍스트 스위칭(context switching)**이라고 한다.

![image.png](attachment:267687d5-b8e0-4101-83f5-8005200b192f:image.png)

</aside>

## 프로세스의 상태

<aside>

![image.png](attachment:ebf44745-97da-48d3-88a1-d46daa133098:image.png)

### New

프로세스를 생성 중인 상태로, 메모리에 적재되어 PCB를 할당받은 상태이다. New 상태를 거쳐 실행할 준비가 된 프로세스는 준비 상태가 되어 CPU의 할당을 기다린다.

### Ready

프로세스가 스케줄링을 기다리는 준비 상태이다. 준비 큐에서 대기하다가 스케줄링되면 Running 상태로 되고 CPU에 의해 실행된다.

### Running

프로세스가 CPU에 의해 현재 실행되고 있는 상태이다. CPU의 시간할당량이 지나면 다시 Ready 상태로 바뀌고 준비 큐에 삽입된다. 프로세스가 입출력을 시행하면 커널은 프로세스를 Blocked 상태로 만들고 대기 큐에 삽입한다.

### Blocked/Wait

프로세스가 자원을 요청하거나, 입출력을 요청하고 완료를 기다리는 상태이다.
입출력이 완료되면 프로세스는 Ready 상태로 바뀌고 준비 큐에 삽입된다.

### Terminated/Zombie

프로세스가 불완전 종료된 상태이다.(좀비 상태)
프로세스가 차지하고 있던 메모리와 할당받았던 자원들은 모두 커널에 의해 반환된 상태이다.
하지만 프로세스 테이블의 항목과 PCB가 여전히 시스템에서 제거되지 않은 상태이다. 이는 프로세스가 남긴 종료 코드(PCB에 있음)를 부모 프로세스가 읽어가지 않아 완전히 종료되지 않은 상태이다.

### Terminated/Out

프로세스가 종료하면서 남긴 종료 코드(PCB에 있음)를 부모 프로세스가 읽어 가서 완전히 종료된 상태이다.
프로세스 테이블의 항목과 PCB가 시스템에서 완전히 제거된 상태이다.

</aside>

## 멀티프로세스와 멀티스레드

<aside>

**멀티프로세스(multi-process)**는 크롬 탭을 여러 개 실행하는 것과 같이 여러 프로세스가 실행되는 것이다.
**멀티스레드(multi-thread)**는 프로세스를 동시에 실행하는 여러 스레드이다.

멀티프로세스의 멀티스레드의 큰 차이는 자원의 공유 여부이다. 프로세스와 스레드의 성질에 따라서 멀티프로세스는 서로의 자원을 공유하지 않지만 멀티스레드는 서로의 자원을 공유한다.

</aside>

## 프로세스 간 통신

<aside>

프로세스는 기본적으로 자원을 공유하지 않지만, 프로세스 간에도 자원을 공유하고 데이터를 주고 받을 수 있는 방법이 있다. 이를 **프로세스 간 통신(Inter-Process Communication, IPC)**라고 한다.

IPC에는 크게 2가지 유형 공유 메모리, 메시지 전달이 있다.

### 공유메모리

프로세스 간에 공유하는 메모리 영역을 토대로 데이터를 주고받는 통신 방법이다.

![image.png](attachment:a5dd5b31-6c0b-4b00-a1b2-e97b91202798:image.png)

마치 자신의 메모리 영역을 읽고 쓰는 것처럼 통신한다. 
데이터를 주고받는 과정에서 커널의 개입이 거의 없다.
각각의 프로세스가 서로의 공유 메모리 영역을 동시에 읽고 쓸 경우 **레이스 컨디션**이 발생할 수 있다.

### 메시지 전달

프로세스 간에 주고받을 데이터가 커널을 거쳐 송수신되는 통신 방식이다.
메시지 잔달 기반 IPC를 위한 대표적인 수단에는 파이프, 시그널, 소켈, 원격 프로시저 호출(RPC)등이 있다.

**파이프(pipe)**란 단방향 프로세스 간의 통신 도구이다. 한쪽으로만 데이터를 전송할 수 있고 읽기용 파이프와 쓰기용 파이프 2개를 이용해 양방향으로 통신하는 경우가 많다.

![image.png](attachment:f0cd9a1f-8326-4142-87ea-d7559bdef295:image.png)

**시그널(signal)**은 프로세스에게 특정 이벤트가 발생했음을 알리는 비동기적인 신호이다.
프로세스는 시그널이 발생하면 시그널 처리를 위한 시그널 핸들러(signal handler)를 실행한 뒤 실행을 재개한다.
**코어 덤프(core dump)**란 주로 비정상적으로 종료하는 경우에 생성되는 파일로, 프로그램이 특정 시점에 작업하던 메모리 상태가 기록되어있다.

| 시그널 | 설명 | 기본 동작 |
| --- | --- | --- |
| SIGCHLD | 자식 프로세스 종료 | 무시 |
| SGILL | 허용하지 않은 명령어 실행 | 코어 덤프 생성 후 종료 |
| SIGINT | 키보드 인터럽트(ctrl + c) | 종료 |
| SIGKILL | 프로세스 종료(핸들러 재정의 불가능) | 종료 |
| SIGSEGV | 잘못된 메모리 접근 | 코어 덤프 생성 후 종료 |
| SIGTERM | 프로세스 종료(핸들러 재정의 가능) | 종료 |
| SIGUSR1 | 사용자 정의 시그널 | 종료 |
| SIGUSR2 | 사용자 정의 시그널 | 종료 |
</aside>
