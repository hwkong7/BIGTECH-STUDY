# 2. 프로세스와 스레드

# 사용자 영역

## 정적 할당 영역

## 1. 코드 영역(텍스트 영역)

: 실행 가능한 명령어가 저장되는 공간

- 읽기 전용 공간

## 2. 데이터 영역

: 실행되는 동안 유지할 데이터가 저장되는 공간

- 정적 변수
- 전역 변수

## 동적 할당 영역

## 3. 힙 영역

: 개발자가 직접 할당 가능한 저장 공간

- 메모리 누수
  - 해결 ⇒ 가비지 컬렉션

## 4. 스택 영역

: 데이터 역역에 담기는 값과는 달리 일시적으로 사용할 값들이 저장되는 공간

- 매개변수
- 지역 변수
- 함수 복귀 주소

<aside>
🗣

**스택 VS 힙 차이**

| **구분**      | **스택 (Stack)**              | **힙 (Heap)**                     |
| ------------- | ----------------------------- | --------------------------------- |
| **관리 주체** | OS/컴파일러 (자동)            | 개발자 (수동/가비지 컬렉터)       |
| **할당 시점** | 컴파일 타임 (크기 고정)       | 런타임 (동적 할당)                |
| **할당 방향** | 높은 주소 → 낮은 주소         | 낮은 주소 → 높은 주소             |
| **장점**      | 매우 빠름 (CPU 레지스터 관리) | 용량 제한이 적고 자유로움         |
| **단점**      | 공간 제한적, 오버플로우 위험  | 상대적으로 느림, 메모리 누수 위험 |

</aside>

# 프로세스의 상태

| 상태                   | 설명                                                                        |
| ---------------------- | --------------------------------------------------------------------------- |
| 생성 상태 (new)        | 프로세스가 생성되는 단계로, PCB가 만들어지고 메모리에 적재 준비를 하는 상태 |
| 준비 상태 (ready)      | CPU를 할당받기 위해 대기하는 상태 (언제든 실행 가능)                        |
| 실행 상태 (running)    | CPU를 할당받아 실제로 실행 중인 상태                                        |
| 대기 상태 (blocked)    | 입출력(I/O) 작업 등 이벤트를 기다리는 상태 (CPU 사용 불가)                  |
| 종료 상태 (terminated) | 프로세스 실행이 완료되어 종료된 상태                                        |

# 멀티프로세스와 멀티스레드

## 멀티프로세스

- 동시에 여러 프로세스가 실행되는 것

![크롬의 멀티프로세스](image%203.png)

크롬의 멀티프로세스

<aside>
❗

- 다른 프로세스들이 기본적으로 **자원을 공유 X**
- 독립적으로 실행
- 같은 작업을 수행하고 있지만 각각의 PID(프로세스ID)값이 다름
- **자원이 독립적으로 할당 ⇒ 다른 프로세스 영향 X**
</aside>

## 멀티스레드

- 프로세스를 동시에 실행하는 여러 스레드
- 하나의 스레드는 스레드를 식별할 고유 정보인 스레드ID, 프로그램 카운터, 레지스터 값, 스택 등으로 구성
- 각각의 프로그램 카운터 값과 스택을 가짐 ⇒ 다음에 실행할 주소 가짐
- ⇒ 연산 과정의 임시 저장 값을 가짐

![image.png](image%204.png)

![image.png](image%205.png)

<aside>
❗

- 동일한 주소 공간의 코드, 데이터, 힙 영역을 공유
- 자원을 공유 ⇒ 협력&통신
- **같은 프로세스를 실행하는 여러 스레드들은 프로세스의 자원을 공유 O**
- 한 스레드에 문제가 생김 ⇒ 전체의 문제가 됨
  ![image.png](image%206.png)

</aside>

<aside>
🗣

**멀티프로세스 vs 멀티스레드 차이**

| **구분**      | **멀티프로세스 (Multi-Process)**        | **멀티스레드 (Multi-Thread)**             |
| ------------- | --------------------------------------- | ----------------------------------------- |
| **핵심 가치** | **안정성** (독립적 구조)                | **효율성** (자원 공유)                    |
| **장점**      | 하나가 죽어도 다른 프로세스에 영향 없음 | 자원 소모 적음, 통신(IPC)이 빠름          |
| **단점**      | 자원 소모 큼, 문맥 교환 느림            | 동기화 문제 발생, 하나가 죽으면 전체 영향 |
| **예시**      | 웹 브라우저 탭, PBP 모델                | 웹 서버 내 작업 처리, TBP 모델            |

</aside>

## 메시지 전달

<aside>
💡

**메시지 전달** : 프로세스 간에 주고받을 데이터를 메시지의 형태로 주고받는 방식

</aside>

<aside>
❗

- send(), recv() 시스템 콜 호출 ⇒ 송수신
- 커널을 통해 송수신 ⇒ 공유 메모리 기반 IPC보다 통신 속도 느림
- 파이프, 시그널, 소켓, 원격 프로시저 호출(RPC) - 파이프
          ![image.png](image%208.png)

          - 익명 파이프(unnamed pipe)
              - 단방향
              - 부모 - 자식 간

          - 지명 파이프(named pipe)
              - 양방향
              - 임의의 프로세스 간
      - 시그널 : 프로세스에게 특정 이벤트가 발생했음을 알리는 **비동기적** 신호


          | 시그널 | 설명 | 기본 동작 |
          | --- | --- | --- |
          | SIGCHLD | 자식 프로세스 종료 | 무시 |
          | SIGILL | 허용하지 않은 명령어 실행 | 코어 덤프 생성 후 종료 |
          | SIGINT | 키보드 인터럽트 (Ctrl + C) | 종료 |
          | SIGKILL | 프로세스 종료 (핸들러 재정의 불가능) | 종료 |
          | SIGSEGV | 잘못된 메모리 접근 | 코어 덤프 생성 후 종료 |
          | SIGTERM | 프로세스 종료 (핸들러 재정의 가능) | 종료 |
          | SIGUSR1 | 사용자 정의 시그널 | 종료 |
          | SIGUSR2 | 사용자 정의 시그널 | 종료 |
          - 시그널 핸들러
          - ⇒ 직접적으로 메시지 주고받X / 비동기적으로 원하는 동작 수행 O
  </aside>
