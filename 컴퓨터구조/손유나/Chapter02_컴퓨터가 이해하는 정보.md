## 정보의 단위

- 비트(bit)
    - 0과 1을 나타내는 가장 작은 정보의 단위
    - N비트 → `$2^N$`개의 정보 표현 가능
- 바이트(byte)
    - `8bit`를 묶은 단위
    - bit는 프로그램 크기에 비해 너무 작은 단위이므로 byte를 사용
    - 프로그램의 관점에서 본 정보의 단위 == byte
        
        ![IMG_E31F6986311F-1.jpeg](attachment:aea81cbe-c39e-4f3c-a0b0-1c8e0c77766a:IMG_E31F6986311F-1.jpeg)
        
- 워드(word)
    - CPU가 한 번에 처리할 수 있는 데이터의 크기
    - 프로그램의 크기가 2GB라고 해서 CPU도 한 번에 2GB를 읽어 들여 처리하지 X
        - CPU는 프로그램을 워드 단위로 읽어 들이고 처리

## 데이터

- CPU는 컴퓨터 내부에서 2진법을 사용해 0과 1만을 이해
- but 2진법을 사용하면 표현하는 숫자의 길이가 너무 길어짐 → 컴퓨터가 이해하는 정보를 표현할 때 2진수와 더불어 16진수도 함께 사용
    - 16진수의 사용 예시 → MAC 주소, IPv6 주소

## 컴퓨터의 소수 표현

- 표현하고자 하는 소수와 실제로 저장된 소수 간에는 오차가 존재할 수 있음.
- 오차 발생 이유
    - 컴퓨터 내부에서는 소수점을 나타내기 위해 부동 소수점 표현 방식을 이용하는데, 이 방식의 정밀도에 한계가 있기 때문
    - **부동 소수점**: 소수점이 고정되어 있지 않은 소수 표현 방식
        - 소수점의 위치를 고정하지 않고도 같은 소수를 다양하게 표현 가능
        - `123.123` == `1.23123 x $10^2$` 또는 `1231.23 x $10^{-1}$`
        - 지수: `2`, `-1`
        - 가수: `1.23123`, `1231.23`
- 2진수 체계에서는 소수를 이와 유사하게 `m x $2^n$`의 꼴로 나타냄.
- 대부분의 컴퓨터는 2진수의 지수와 가수를 아래와 같은 형식으로 저장 (IEEE 754)
    
    ![IMG_B31BB920EACC-1.jpeg](attachment:a7d2835d-b0b8-4477-82e6-9a03f15f2808:IMG_B31BB920EACC-1.jpeg)
    
    - 부호 비트가 `0`이면 양수, `1`이면 음수를 의미
- 지수는 어떻게 저장하는가?
    - 지수에 바이어스(bias)값을 더해서 저장
    - 바이어스 값 = `$2^{k-1}-1$`(`k`는 지수의 비트수로 `8`비트 혹은 `11`비트가 됨)
- 가수는 어떻게 저장하는가?
    - 가수는 `1.OOO…`의 형태를 띔. → 따라서 컴퓨터가 가수를 저장할 때는 가수인 `1.OOO`에서 `1`을 제외한 `OOO`에 해당하는 소수 부분만 저장
- 10진수 소수를 2진수로 표현할 때, 10진수 소수와 2진수 소수의 표현이 딱 맞아떨어지지 않을 수 있음.
    - 10진수 `0.1`은 `m x $10^n$`의 꼴로 간단하게 나타낼 수 있지만, 같은 수를 `1.m x $2^n$`의 꼴로 표현하려면 무한하게 많은 소수점이 필요
    - 컴퓨터의 저장공간은 한정적이기 때문에 무한히 많은 소수점 저장 불가능 → 따라서 딱 맞아떨어지지 않는 소수를 표현할 때는 일부 소수점을 생략하여 저장하기 때문에 오차가 발생하게 되는 것임.

## 0과 1로 문자 표현하기

- 문자 집합: 컴퓨터가 이해할 수 있는 문자들의 집합
- 문자 인코딩: 문자 집합에 속한 문자를 컴퓨터가 이해하는 `0`과 `1`로 이루어진 문자 코드로 변환하는 과정
- 문자 디코딩: `0`과 `1`로 표현된 문자를 사람이 이해하는 문자로 변환하는 과정
- 글자 깨짐 현상의 원인 → 웹사이트가 특정 인코딩 방법을 지원하지 않거나, 인코딩된 문자를 디코딩하는 방법을 알지 못하기 때문
    
    ![IMG_161231EAD21B-1.jpeg](attachment:3af1b979-d605-4a5f-a90c-980e5b83621d:IMG_161231EAD21B-1.jpeg)
    

**아스키**

- 가장 기본적인 문자 집합
- 하나의 아스키 문자를 표현하기 위해선 `8bit`를 사용
    - 이 중 `1bit`는 패리티 비트로, 이는 오류 검출을 위해 사용됨.
    - `7bit`로 문자 표현 → 총 `128`개의 문자 표현 가능
- 아스키 문자들은 `0`부터 `127`까지의 숫자 중 하나의 고유한 수에 대응됨.
- 아스키 코드: 아스키 문자에 대응된 고유한 수
- 아스키 코드의 인코딩 방식
    - 아스키 코드를 2진수로 표현함으로써 아스키 문자를 `0`과 `1`로 대응시킴.
- 코드 포인트: 문자 인코딩에서 글자에 부여된 고유한 값
- 아스키 코드는 한글 표기 불가능

**EUC-KR**

- 한글 인코딩 방식 중 하나
- KS X 1001, KS X 1003이라는 문자 집합 기반의 인코딩 방식
- 아스키 문자 표현 → `1`byte
- 하나의 한글 글자 표현 → `2`byte
    - 2byte(16bit)는 4자리 16진수로 표현 가능하므로 EUC-KR로 인코딩된 한글 글자 하나는 4자리 16진수로 나타낼 수 있음.
- 아스키 코드에 비해 표현할 수 있는 문자가 많지만, 아직도 모든 한글 조합을 표현할 수는 없음.

**유니코드**

- 한글을 포함해 EUC-KR에 비해 훨씬 많은 언어, 특수문자, 화살표, 이모티콘까지 코드로 표현할 수 있는 통일된 문자 집합
- 유니코드가 없었다면 각각의 언어마다 다른 문자 집합과 인코딩 방식을 이해해야함. → but 유니코드가 대부분의 언어를 지원하기 때문에 국가별로 다른 문자 집합과 인코딩 방식을 준비할 필요가 없어짐.
- 유니코드는 아스키코드나 EUC-KR과 달리 글자에 부여된 값 자체를 인코딩 값으로 삼지 않고, 이 값을 UTF-8, UTF-16, UTF-32 등의 방법으로 인코딩함.
    - UTF-8, UTF-16, UTF-32: 유니코드 문자에 부여된 값을 인코딩하는 방식
- UTF-8, UTF-16: 가변 길이 인코딩 방식
    - 가변 길이 인코딩 방식: 인코딩된 결과의 길이가 일정하지 않을 수 있음.
- UTF-32: 고정 길이 인코딩 방식

**base64**

- 비단 문자뿐만 아니라, 이진 데이터까지 변환할 수 있는 인코딩 방식
- 이미지 등 단순 문자 이외의 데이터까지 모두 아스키 문자 형태로 표현 가능
- 64진수 하나를 표현하기 위해서는 `$2^6$`의 지수인 `6bit`가 필요 → 따라서 변환할 데이터를 6bit씩 나누어 하나의 문자로 변환(기본적으로 4개씩(24bit) 한 번에 변환됨.)
- 패딩: 나누어 떨어지지 않는 자리를 채우는 의미없는 `0`

## 명령어

- 수행할 동작
- 수행할 대상
    - 수행할 동작에 사용될 **데이터 자체**
    - 수행할 동작에 사용될 **데이터가 저장된 위치**
- 연산 코드
    - 명령어가 수행할 동작
    - 연산 코드 필드에 담김
- 오퍼랜드
    - 동작에 사용될 데이터 또는 동작에 사용될 데이터가 저장된 위치(ex. 메모리나 레지스터의 주소)
    - 오퍼랜드 필드에 담김
- 하나의 명령어는 연산 코드와 0개 이상의 오퍼랜드로 구성됨

## 기계어와 어셈블리어

- 기계어: CPU가 이해할 수 있도록 `0`과 `1`로 표현된 정보를 있는 그대로 표현한 언어
- 어셈블리어: `0`과 `1`로 표현된 기계어를 읽기 편한 형태로 단순 번역한 언어
- CPU마다 구체적인 연산 코드의 종류나 레지스터의 이름, 명령어의 생김새가 다를 수 있음.
    - 어셈블리어는 기계어를 그대로 번역한 언어이기 때문에 명령어의 종류와 생김새가 다르면 기계어도 달라지고, 이를 번역한 어셈블리어도 달라지게 됨.
- 같은 프로그램일지라도 CPU마다 이해하는 명령어가 다르면 실행 불가능
    - ex. 인텔 CPU를 사용하는 컴퓨터에서 만들어진 실행 파일은 애플 CPU를 사용하는 컴퓨터에 그대로 옮겨 실행 불가능

## 명령어 사이클

> CPU가 명령어를 처리하는 과정에서 프로그램 속 각각의 명령어들이 반복되며 실행되는 주기
> 
- 과정
    - 인출 사이클(fetch cycle): 메모리에 있는 명령어를 CPU로 가지고 오는 단계
    - 실행 사이클(execution cycle): CPU로 가져온 명령어를 실행하는 단계
    - 인출 사이클과 실행 사이클을 반복하며 프로그램을 실행
- 명령어를 인출했더라도 곧바로 실행할 수 없는 경우 → 오퍼랜드 필드에 메모리 주소가 명시된 경우
- 간접 사이클(indirect cycle): 명령어를 실행하기 위해 한 번 더 메모리에 접근하는 단계
