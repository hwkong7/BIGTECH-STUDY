# 레지스터

- CPU 안에 있는 작은 임시 저장장치

## 1 프로그램 카운터(PC, Program Counter)

- 다음에 읽어 올 **명령어의 주소**를 저장
- = 명령어 포인터(IP, Instruction Pointer)

## 2 명령어 레지스터(IR, Instruction Register)

- 방금 읽어 온 **명령어 자체**를 저장하는 레지스터
- 작동 흐름
  - 제어장치는 IR의 명령어를 해석한 뒤
    - → ALU에 연산을 시키거나
    - → 다른 장치에 제어 신호를 보낸다

## 3 범용 레지스터(general purpose register)

- 다양한 용도로 자유롭게 사용하는 레지스터
- **데이터 / 명령어 / 주소**를 저장할 수 있음

## 4 플래그 레지스터(flag register)

- 연산 결과나 CPU 상태를 나타내는 **플래그 값** 저장
- 플래그는 CPU가 판단에 참고하는 **상태 비트**
- 대표적인 플래그의 종류
  | 종류              | 의미                         |
  | ----------------- | ---------------------------- |
  | 부호 플래그       | 결과가 양수인지 음수인지     |
  | 제로 플래그       | 결과가 0인지                 |
  | 캐리 플래그       | 올림수·빌림수 발생 여부      |
  | 오버플로우 플래그 | 오버플로우 발생 여부         |
  | 인터럽트 플래그   | 인터럽트 허용 여부           |
  | 슈퍼바이저 플래그 | 커널 모드 / 사용자 모드 여부 |

## 5 스택 포인터

- 메모리 내 스택 영역의 최상단 스택 데이터 위치를 가르키는 특별한 레지스터

# 인터럽트

- CPU의 작업을 방해하는 신호

## 동기 인터럽트(synchronous interrupts)

- CPU에 의해 발생하는 인터럽트
- CPU가 프로그래밍 오류와 같은 예외적인 상황(예상치 못한 상황)을 마주쳤을 때 발생하는 인터럽트 ⇒ = 예외

## 비동기 인터럽트(asynchronous interrupts)

- 입출력장치에 의해 발생하는 인터럽트
- ≒ 세탁기의 완료 알림, 전자레인지의 조리 완료 알림
- ex) 키보드 입력, 프린터 작업 완료
- = 하드웨어 인터럽트

## 하드웨어 인터럽트

- 효율적으로 명령어를 처리하기 위해 사용하는 인터럽트
- CPU가 입출력 완료 여부를 계속 확인하지 않아도 되게 해 주는 방식
- 즉, CPU가 쓸데없이 기다리지 않아도 되므로 **효율적**이다

### 폴링(polling)

- 작업 완료 여부를 계속해서 확인하는 방식
- 하드웨어 인터럽트를 쓰지 않으면 비효율적일 수 있음

### CPU가 하드웨어 인터럽트를 처리하는 순서

1. 입출력장치가 CPU에 **인터럽트 요청 신호**를 보낸다.
2. CPU는 명령어 실행이 끝날 때마다 인터럽트 여부를 확인한다.
3. 인터럽트 플래그를 보고 받아들일 수 있는지 판단한다.
4. 가능하면 현재 작업 내용을 **백업**한다.
5. 인터럽트 벡터를 참고해 **인터럽트 서비스 루틴(ISR)** 을 실행한다.
6. 실행이 끝나면 백업한 내용을 복구하고 원래 작업으로 돌아간다.

## 인터럽트 요청 신호

- 인터럽트하기 전에 CPU에게 인터럽트 가능 여부를 확인하기 위해 보내는 신호

## 인터럽트 플래그(interrupt flag)

- CPU가 하드웨어 인터럽트를 받아들일지, 무시할지를 결정하는 플래그
- 인터럽트 플래그가 활성화 ⇒ CPU가 인터럽트 요청을 수용
- 인터럽트 플래그가 비활성화 ⇒ CPU는 인터럽트 요청이 오더라도 해당 요청을 무시

### 막을 수 있는 인터럽트(maskable interrupt)

- 인터럽트 플래그로 막을 수 있는 인터럽트

### 막을 수 없는 인터럽트(Non-Maskable Interrupt = NMI)

- 인터럽트 플래그로 막을 수 없는 인터럽트
- 가장 우선순위가 높은, 즉 가장 먼저 처리해야 하는 인터럽트
- 정전이나 하드웨어 고장 등과 관련된 인터럽트가 이에 해당한다.

![image.png](attachment:2e5ed8be-713d-4229-9412-c95f286d450c:118137e2-637a-4ab9-a42c-50742da8eb9b.png)

## 인터럽트 서비스 루틴(ISR, Interrupt Service Routine)

- 인터럽트를 처리하기 위한 프로그램
- = 인터럽트 핸들러(interrupt handler)
- CPU가 인터럽트를 처리한다는 것은
- ⇒ 결국 해당 ISR을 실행한다는 뜻

![image.png](attachment:5bf2350b-35d1-46f0-a5ef-93feff5b7c18:bc7b9272-701d-4123-9642-053bb08c4cec.png)

![image.png](attachment:c04b8c34-ee31-4fb8-bf58-c9ee6daf36c1:image.png)

## 인터럽트 벡터(interrupt vector)

- 인터럽트 서비스 루틴을 식별하기 위한 정보

![image.png](attachment:34cc3ff1-89f1-40f4-b535-8f6e1532c82c:image.png)

- CPU는 수많은 인터럽트 서비스 루틴을 구분하기 위해 인터럽트 벡터를 이용한다.
- CPU는 하드웨어 인터럽트 요청을 보낸 대상으로부터 버스를 통해 인터럽트 벡터를 전달받는다.
- 인터럽트 벡터가 인터럽트 서비스 루틴의 시작 주소를 포함하고 있기 때문에 CPU는 특정 인터럽트 서비스 루틴을 실행할 수 있다.

## 인터럽트 처리 시 백업되는 내용

- ISR도 레지스터와 스택을 사용하므로
- CPU는 원래 작업을 재개할 수 있도록 필요한 값을 스택에 저장한다
- 예: 프로그램 카운터, 스택 포인터 등

![image.png](attachment:66a0322c-599c-4410-bde9-bde7ddb104f6:image.png)

## 인터럽트 사이클

- CPU는 명령어 사이클만 반복하는 것이 아니라 필요할 때 **인터럽트 사이클**도 수행한다

![image.png](attachment:4d23e323-edb8-46e9-98f9-762b19ed8559:image.png)

# 예외

- 동기 인터럽트의 한 종류
- CPU는 예외가 발생하면 하던 일을 중단하고 해당 예외를 처리한다.
- 예외를 처리하고 나면 다시 본래 하던 작업으로 되돌아와 실행을 재개한다.

![image.png](attachment:77409641-794d-435d-a0bf-c6cb6833abb2:image.png)

## 예외의 종류

- 폴트(fault)
- 트랩(trap)
- 중단(abort)
- 소프트웨어 인터럽트(software interrupt)

## 폴트(fault)

- 예외를 처리한 직후에 **예외가 발생한 명령어부터 실행**을 재개하는 예외
- 대표적인 예) 페이지 폴트

![image.png](attachment:32f12548-c161-492c-ad72-6d3b1aadb788:image.png)

## 트랩(trap)

- 예외를 처리한 직후에 예외가 발생한 명령어의 **다음 명령어부터 실행**을 재개하는 예외

## 중단(abort)

- CPU가 실행 중인 프로그램을 강제로 중단시킬 수밖에 없는 **심각한 오류**를 발견했을 때 발생하는 예외

## 소프트웨어 인터럽트(software interrupt)

- **시스템 콜**이 발생했을 때 발생하는 예외
- 운영체제가 제공하는 기능을 사용하기 위해 프로그램이 **의도적으로 발생**시키는 인터럽트라고 볼 수 있다.

# CPU 성능 향상을 위한 설계

## CPU 성능을 높이는 대표 요소

- 클럭 속도
- 코어 수
- 스레드 수
- 파이프라이닝

## CPU 클럭 속도

- 클럭(clock)이란 컴퓨터의 부품을 일사불란하게 움직일 수 있게 하는 시간의 단위
- 단위는 **Hz**

$$
1\mathrm{GHz} = 1000000000(10^9)\mathrm{Hz}
$$

![image.png](attachment:b952b9ac-d21d-4b17-b67e-22877c4b6a40:image.png)

### 클럭 속도와 CPU 성능

- 클럭 속도가 높을수록 명령어 처리 속도는 빨라질 수 있음
- 하지만 발열과 전력 소모 문제가 있으므로
  ⇒ 클럭만 높인다고 성능이 무한히 좋아지지는 않음

# 멀티코어와 멀티스레드

## 코어(core)

- CPU 내에서 **명령어를 읽어 들이고, 해석하고, 실행하는 부품**
- 요즘 CPU는 이런 코어가 여러 개 들어 있음

![image.png](attachment:97868ebb-eae3-4b9c-9610-d060e9cf37be:image.png)

## 멀티코어 CPU

- 여러 개의 코어를 포함하고 있는 CPU
- = 멀티코어 프로세서

                                   멀티코어

| 코어 개수 | 명칭                    |

| 1         | 싱글코어(single-core)   |

| 3         | 트리플코어(triple-core) |

## 스레드(thread)

- 실행 흐름의 단위

### 하드웨어 스레드

- CPU에서 사용하는 스레드
- 하나의 코어가 동시에 처리하는 명령어의 단위
- **논리 프로세서(logical processor)** 라고도 부름

![image.png](attachment:a7fab7f4-fe3f-43c1-97b2-b9c04d57fd6b:image.png)

### 소프트웨어적 스레드(소프트웨어 스레드)

- 하나의 프로그램에서 독립적으로 실행되는 단위
- 하나의 프로그램 안에서도 여러 스레드가 독립적으로 실행될 수 있음

![image.png](attachment:bab66f29-2ecc-43fc-89ec-80f80f06d771:image.png)

## 하드웨어 스레드 vs 소프트웨어 스레드

- **하드웨어 스레드** → CPU 관점의 실행 단위
- **소프트웨어 스레드** → 프로그램 관점의 실행 단위
- 하드웨어 스레드는 ‘병렬성’을 구현하기 위한 물리적인 실행 단위에 가깝고, 소프트웨어 스레드는 ‘동시성’을 구현하기 위한 논리적인 실행 단위에 가깝다.

## 병렬성(parallelism)

- 작업을 물리적으로 동시에 처리하는 성질
- 같은 시점에 여러 작업이 실제로 동시에 처리되는 것
- 보통 여러 코어나 여러 하드웨어 자원이 필요함

## 동시성(concurrency)

- 동시에 작업을 처리하는 것처럼 보이는 성질
- 실제로는 CPU가 매우 빠르게 여러 작업을 번갈아 가며 처리하기 때문에 사용자 눈에는 동시에 처리되는 것처럼 보인다.
- 물리적으로 같은 시점에 여러 작업이 동시에 처리되는 것은 아니다.

# 파이프라이닝을 통한 명령어 병렬 처리

## 명령어 병렬 처리 기법(ILP, Instruction-Level Parallelism)

- 여러 명령어를 동시에 처리하여 CPU를 한시도 쉬지 않고 작동시킴으로써 CPU의 성능을 높이는 기법
- 대표적인 방법이 **명령어 파이프라이닝**

## 명령어 파이프라이닝(instruction pipelining)

- 명령어를 처리하는 과정을 비슷한 시간 간격으로 나누고, 각 단계를 동시에 실행하는 기법

### 일반적인 명령어 처리 단계

1. 명령어 인출(Instruction Fetch)
2. 명령어 해석(Instruction Decode)
3. 명령어 실행(Execute Instruction)
4. 결과 저장(Write Back)

![image.png](attachment:e1f95da2-3cb9-46b8-a10b-6ed83c16453e:image.png)

[그림]

### 파이프라이닝의 핵심

- 같은 단계가 겹치지만 않는다면 CPU가 각각의 단계를 동시에 실행할 수 있다.
- 즉, 한 명령어가 인출되는 동안 다른 명령어는 해석될 수 있고, 또 다른 명령어는 실행될 수 있다.
- 이처럼 여러 명령어를 겹쳐 실행하여 성능을 높일 수 있다.

## 슈퍼스칼라(superscalar)

- CPU 내부에 여러 명령어 파이프라인을 포함하는 구조
- 파이프라인이 여러 개라 더 많은 명령어를 병렬 처리할 수 있다

![image.png](attachment:7798ad8d-9692-4aeb-953f-cdccf69573cc:image.png)

## 명령어 집합과 파이프라인 위험

### CISC(Complex Instruction Set Computer)

- 다채로운 기능을 지원하는 복잡한 명령어들로 구성된 명령어 집합
- 적은 수의 명령어로도 작업 가능
- 하지만 명령어 길이와 실행 시간이 일정하지 않아
  → 파이프라이닝에 불리할 수 있다.

![image.png](attachment:22efdc5b-6593-4c49-8a41-c70091de6ef0:image.png)

### RISC(Reduced Instruction Set Computer)

- CISC에 비해 활용 가능한 명령어의 종류가 적은 명령어 집합
- 보통 짧고 실행 시간이 비교적 일정하다
- 같은 프로그램이라도 CISC보다 더 많은 명령어가 필요할 수 있다.
- 대신 명령어의 길이와 수행 시간이 비교적 규칙적 ⇒ 파이프라이닝에 최적화되어 있다.

![image.png](attachment:26055061-e823-44a9-8e57-df1e36d9d0af:image.png)

### CISC와 RISC 비교

| 구분         | CISC                | RISC                        |
| ------------ | ------------------- | --------------------------- |
| 명령어 형태  | 복잡하고 다양함     | 단순하고 규격화됨           |
| 명령어 수    | 적은 수로 작업 가능 | 더 많은 수가 필요할 수 있음 |
| 실행 시간    | 일정하지 않은 편    | 비교적 일정                 |
| 파이프라이닝 | 불리할 수 있음      | 유리함                      |

### 파이프라인 위험(pipeline hazard)

- 파이프라이닝이 실패하여 성능 향상이 이루어지지 않는 상황
- 크게 데이터 위험, 제어 위험, 구조적 위험으로 구분할 수 있다.

### 데이터 위험(data hazard)

- 명령어 간의 데이터 의존성에 의해 발생하는 위험
- 어떤 명령어는 이전 명령어의 실행 결과가 있어야만 실행될 수 있다.
- 예를 들어 어떤 명령어가 계산 결과를 레지스터에 저장하기 전에, 다음 명령어가 그 값을 사용하려 한다면 문제가 발생한다.

### 예시

```sql
명령어 1: R1 ← R2 + R3
명령어 2: R4 ← R1 + R5
```

- 명령어 2는 명령어 1의 결과인 R1 값이 먼저 저장되어야 실행할 수 있으므로 데이터 위험이 발생할 수 있다.

### 제어 위험(control hazard)

- 프로그램 카운터 값의 갑작스러운 변화에 의해 발생하는 위험
- 분기, 점프, 인터럽트 등으로 실행 흐름이 바뀌는 경우 발생
  ⇒ 미리 가져온 명령어가 쓸모없어질 수 있다.

### 구조적 위험(structural hazard)

- 명령어들을 겹쳐 실행하는 과정에서 서로 다른 명령어가 동시에 ALU, 레지스터 등 같은 CPU 자원을 사용하려고 할 때 발생하는 위험
- = 자원 위험(resource hazard)
