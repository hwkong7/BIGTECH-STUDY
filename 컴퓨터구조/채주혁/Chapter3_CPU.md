# 레지스터

> CPU 안에 있는 작은 임시 저장장치
> 

## 프로그램 카운터

- 메모리에서 다음에 읽어 들일 명령어의 주소
- 1씩 증가 → 다음으로 읽을 메모리 주소가 1씩 증가하는 것과 같음
    - 따라서 메모리에 저장된 프로그램이 순차적으로 실행 가능
- 조건이나 리턴문을 실행하는 경우 순차적이지 않고 임의의 위치로 변경

## 명령어 레지스터

- 방금 읽은 명령어를 저장하는 레지스터
- 제어 장치가 해석 후 ALU에게 연산을 시키거나 다른 부품을 작동시킴

## 범용 레지스터

- 일반적인 상황에서 사용할 수 있는 레지스터
- 데이터와 명령어, 주소 모두 저장가능

## 플래그 레지스터

- 연산의 결과 혹은 CPU 상태 부가 정보인 플래그를 저장
- 플래그
    - CPU가 명령을 처리하는 과정에서 반드시 참조해야 할 상태정보

## 스택 포인터

- 실행 중인 프로그램들이 적재
- 스택 형태로 사용 가능한 주소 공간을 하나 이상 가지고 있음
- 메모리 내 스택 영역의 최상단 스택 데이터 위치를 가리키는 레지스터

# 인터럽트

> CPU의 작업을 방해하는 신호
> 
- 동기 인터럽트
    - CPU에 발생하는 인터럽트
    - 예외적인 상황
- 비동기 인터럽트
    - 입출력 장치에 의해 발생하는 인터럽트
    - 하드웨어 인터럽트

## 하드웨어 인터럽트

> 명령어를 효율적으로 처리하기 위해 사용
> 
- 프린트가 진행되는 동안 다른 작업을 처리
    - 입출력 완료 여부를 확인하기 위한 CPU 사이클 낭비 최소화
    - 다른 일을 수행할 수 있는 시간을 벌어줌
1. 입출력장치가 CPU에게 인터럽트 요청 신호
2. CPU는 명령어를 인출하기 전 인터럽트 여부 확인
3. 인터럽트 요청 확인 → 인터럽트 플래그를 통해 현재 인터럽트를 받을 수 있는지 확인
4. 받을 수 있다면 CPU가 지금의 작업을 백업
5. CPU는 인터럽트 벡터를 참고하여 인터럽트 서비스 루틴을 실행
6. 인터럽트 서비스 루틴이 끝나면 4에서 백업해 둔 작업을 복구하여 실행을 재개

- 인터럽트 요청 신호
    - CPU에게 인터럽트 가능 여부를 물어봄
- 인터럽트 플래그
    - CPU가 인터럽트 요청을 수용하기 위해서 플래그가 활성화되어있어야 함
    - 불가능으로 설정되어 있다면 해당 요청을 무시
    - 우선 순위가 높다면 인터럽트 요청을 무시하지 않음
- 인터럽트 벡터
- 
- 인터럽트 서비스 루틴
    - 인터럽트를 처리하기 위한 프로그램
    - 어떤 인터럽트가 발생했을 때 해당 인터럽트를 어떻게 처리하고 작동해야할지
    - 인터럽트 처리한다 → 인터럽트 서비스 루틴을 실행하고 본래 수행하던 작업으로 다시 되돌아온다

# CPU 성능 향상을 위한 설계

## CPU 클럭 속도

> 컴퓨터 부품을 움직일 수 있게 하는 시간의 단위
> 
- 헤르츠
    - 클럭 속도
    - 클럭이 1초에 몇 번 반복되는지
- 클럭 속도가 높아지면 CPU는 명령어 사이클을 더 빠르게 반복
- 클럭속도가 높으면 성능이 좋아짐
    - 발열이 심해질 수 있으므로, 클럭을 높이는 데 한계가 있음

## 멀티코어와 멀티스레드

- 코어
    - CPU 내에서 명령어를 읽고, 해석하고, 실행하는 부품
- 멀티 코어
    - 여러 개의 코어를 포함하고 있음
- 스레드
    - 실행 흐름의 단위
    - 하드웨어 스레드
        - 하나의 코어가 동시에 처리하는 명령어의 단위
        - 논리 프로세서
    - 소프트웨어 스레드
        - 하나의 프로그램에서 독립적으로 실행하는 단위
        - 메모리에 적재된 해당 프로그램을 구성하는 여러 부분이 동시에 실행될 수 있다

![image.png](attachment:c04f89ae-7091-4819-81a7-4c7cb3b97ca8:image.png)

- 병렬성
    - 작업을 물리적으로 동시에 처리하는 성질
    - 스레드가 4개인 CPU가 4개의 명령어를 동시에 실행하는 경우
    - 하드웨어 스레드
- 동시성
    - 동시에 작업을 처리하는 것처럼 보이는 성질
    - 사용자 눈에는 마치 여러 작업이 동시에 처리되는 것처럼 보임
    - 소프트웨어 스레드

# 파이프라이닝을 통한 명령어 병렬 처리

✅ 명령어 병렬 처리 기법

- 여러 명령어를 동시에 처리하여 CPU를 쉬지 않고 작동시키는 방법

![image.png](attachment:f876a2c1-a7a8-46a7-a65b-25be446246e6:image.png)

✅ 명령어 파이프라이닝

1️⃣ 명령어 인출

2️⃣ 명령어 해석

3️⃣ 명령어 실행

4️⃣ 결과 저장

- 명령어 병렬 처리에 유리한 명령어 집합이 있고, 불리한 명령어 집합이 있음
- CISC
    - 인텔 x86
    - 다채로운 기능을 지원하는 복잡한 명령어들로 구성된 명령어 집합
    - 적은 수의 명령어로 프로그램을 실행할 수 있음
    - 명령어가 워낙 복잡하고 다양한 기능을 제공
        - 명령어 크기 및 실행되기까지 시간이 일정하지 않고, 하나의 실행에 여러 클럭 주기가 필요
- RISC
    - M1 CPU
    - 명령어의 종류가 작음 → 짧고 규격화, 1클럭 내외

## 파이프라인 위험

![image.png](attachment:aa1b1055-2fa9-4be9-8391-65330eed570f:image.png)

> 파이프라이닝에 실패하여 성능 향상이 이루어지지 않는 상황
> 
- 데이터 위험
    - 명령어 간의 데이터 의존성에 의해 발생
    - 어떤 명령어는 동시에 처리할 수 없고, 이전 명령어를 끝까지 실행해야만 비로소 실행
    - 명령어 2가 1에 의존적인 상황
- 제어 위험
    - 프로그램 카운터의 갑작스러운 변화
    - JUMP, CONDITIONAL JUMP로 실행 흐름이 바뀌면 미리 인출하거나 해석중인 명령어들은 쓸모가 없어짐
- 구조적 위험
    - 서로 다른 명령어가 동시에 ALU, 레지스터 등 같은 CPU 부품을 사용
