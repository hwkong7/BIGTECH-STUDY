## 포트를 통한 프로세스 식별

> 전송 계층의 주된 목적
> 
- 패킷의 최종 송수신 대상 ⇒ **호스트가 실행하는 프로세스**
- 네트워크 상에서 호스트가 실행하는 프로세스는 어떻게 식별할 수 있는가? ⇒ **포트 번호**를 통해 식별 가능
- 네트워크 패킷을 주고받는 프로세스에는 포트 번호가 할당됨. → [`IP 주소` + `포트 번호`]를 통해 특정 호스트가 실행하는 특정 프로세스 식별 가능

## NAT와 NAPT

**NAT**

> 공인 IP 주소와 사설 IP 주소 간 변환을 위해 사용하는 기술
> 
- 네트워크 내부에서 사설 IP 주소를 사용하는 호스트가 네트워크 외부에 있는 호스트와 패킷을 주고받기 위해서는 공인 IP 주소와 사설 IP 주소 간 변환이 필요함.

**문제점**

- 사설 IP 주소와 공인 IP 주소를 일대일로 변환하면, 사설 IP 주소 수만큼의 공인 IP 주소가 필요

**해결**

- 여러 개의 사설 IP 주소가 적은 수의 공인 IP 주소를 공유 (N:1)
- HOW? → 포트 사용

**NAPT**

> IP 주소 변환 과정에서 변환할 IP 주소의 쌍과 더불어, 포트 번호도 함께 고려하는 포트 기반의 NAT
> 

![image.png](attachment:948f50f0-bfd5-468a-b7b7-8a80d47e86dd:image.png)

- 서로 다른 사설 IP 주소가 같은 공인 IP 주소로 변환되더라도 다른 포트 번호로 변환된다면 네트워크 내부의 호스트 특정 가능

## TCP

> 신뢰할 수 있는 프로토콜이자 연결형 프로토콜
> 
- 상태 관리, 흐름 제어, 오류 제어, 혼잡 제어 제공
- 연결 수립, 종료 과정 O
- 속도: `TCP` < `UDP`
- 헤더의 필드 수: `TCP` > `UDP`

![image.png](attachment:53c84864-cb7b-4b28-8248-734cf0a1f64c:image.png)

**순서 번호 필드**

- TCP 패킷의 올바른 송수신 순서를 보장하기 위해 세그먼트 첫 바이트에 매겨진 번호
- 현재 주고받는 TCP 세그먼트가 송수신하고자 하는 데이터의 몇 번째 바이트에 해당하는지 알 수 있음

**확인 응답 번호 필드**

- 상태 호스트가 보낸 세그먼트에 대한 응답으로, 다음으로 수신하길 기대하는 순서 번호
- 세그먼트가 확인 응답 번호를 포함하고 있음을 알리기 위해 `ACK` 플래그를 `1`로 설정

**일부 제어 비트(플래그비트)**

- `8`bit
- ACK: 세그먼트의 승인을 나타내기 위한 비트
- SYN: 연결을 수립하기 위한 비트
- FIN: 연결을 종료하기 위한 비트

## TCP의 오류 · 흐름 · 혼잡 제어

**재전송을 통한 오류 제어**

- TCP는 언제 잘못 전송된 세그먼트가 있다는 것을 인지하는가?
    - 중복된 `ACK` 세그먼트가 도착했을 때
        
        ![image.png](attachment:79376d02-d1f0-43bf-9bda-4d34a33ca301:image.png)
        
    - 타임아웃이 발생했을 때
        
        ![image.png](attachment:cbd2aac8-28f2-4e8b-8429-7763b366eab7:image.png)
        
        - 타임아웃: 타이머의 카운트다운이 끝난 상황
        - TCP 세그먼트를 송신하는 호스트는 모두 재전송 타이머라는 특별한 값을 유지
        - 호스트는 세그먼트를 전송할 때마다 재전송 타이머를 시작

**흐름 제어**

- 수신 호스트가 한 번에 받아 처리할 수 있을 만큼만 전송하는 것
    - 수신 호스트가 한 번에 받을 수 있는 전송량은 TCP 수신 버퍼의 크기에 의해 결정됨.
    - 수신 버퍼: 수신된 세그먼트가 애플리케이션 프로세스에 의해 읽히기 전에 임시 저장되는 공간
- 주체: 수신 호스트
- 송신 호스트는 수신 호스트가 한 번에 처리할 수 있는 양을 어떻게 알고 보내는가?
    
    ⇒ 윈도우 필드에 수신 호스트가 한 번에 처리할 수 있는 수신 윈도우(`RWND`) 크기가 명시돼있음.
    

**혼잡 제어**

- 혼잡: 많은 트래픽으로 인해 패킷의 처리 속도가 느려지거나 유실될 수 있는 상황
- 주체: 송신 호스트
    - 송신호스트가 주체적으로 얼마나 네트워크가 혼잡한지를 판단할 수 있어야 하고, 판단된 혼잡의 정도에 따라 세그먼트의 전송량을 조절할 수 있어야 함.
- 네트워크 혼잡 여부를 판단하는 기준
    - 중복된 `ACK` 세그먼트가 도착했을 때
    - 타임아웃이 발생했을 때
- 송신 호스트는 혼잡 없이 처리할 수 있을 정도의 양(= 혼잡 윈도우, `CWND`)만큼만 송신

**혼잡 제어 알고리즘**

> 혼잡 윈도우 크기를 연산하는 방법
> 
- 혼잡 윈도우의 크기는 송신 호스트가 직접 계산해야함.
- **AIMD**
    - 세그먼트를 보내고, 그에 대한 응답이 오기까지
        - 혼잡 감지 X → 혼잡 윈도우 -= 1
        - 혼잡 감지 O → 혼잡 윈도우 /= 2
    
    ![image.png](attachment:293a2aea-ffee-49a2-a2a1-f4726e70d281:image.png)
    
    - 특징
        - 톱니 모양으로 변화
- RTT: 패킷을 보내고 그에 대한 응답이 수신되기까지의 시간

## TCP의 종료

> 포 웨이 핸드셰이크(four-way handshake)
> 

![image.png](attachment:15a01f26-943a-4f94-aafd-8cf5cddf42e0:image.png)

- 송수신 호스트가 각자 한 번씩 `FIN`과 `ACK`를 주고받으며 이루어짐.
- 액티브 클로즈: 먼저 연결을 종료하려는 호스트에 의해 수행되는 동작
- 패시브 클로즈: 연결 종료 요청을 받아들이는 호스트에 의해 수행되는 동작

## TCP의 상태 관리

- TCP는 상태를 유지 · 관리하는 프로토콜 → 스테이트풀 프로토콜
    - 상태: 현재 어떤 통신 과정에 있는지를 나타내는 정보
- 연결이 수립되지 않았을 때 주로 활용되는 상태
    - `CLOSED`, `LISTEN`
- 연결 수립 과정에서 주로 활용되는 상태
    - `SYN-SENT`, `SYN-RECEIVED`, `ESTABLISHED`
    
    ![image.png](attachment:e60c0a28-682e-4cf2-85b7-242907dbafcc:image.png)
    
- 연결 종료 과정에서 주로 활용되는 상태
    - `FIN-WAIT-1`, `CLOSE-WAIT`, `FIN-WAIT2`, `LAST-ACK`, `TIME-WAIT`
    
    ![image.png](attachment:ecde2e41-bacb-4797-8aeb-cefa94733fc1:image.png)
    
    - CLOSING 상태: 아직 자신의 `FIN` 세그먼트에 대한 `ACK` 세그먼트를 받지 못했을 때 접어드는 상태
        - 양쪽이 동시에 연결 종료를 요청하고 서로의 종료 응답을 기다릴 경우 발생
