## 원본 데이터 변경 여부(날짜)

### If-Modified-Since

- 원본 자원 변경 여부를 묻는 헤더 (대표적)

### 상황

### 1️⃣ 서버가 요청받은 자원이 변경된 경우

- 서버는 상태 코드 200(OK)과 함께 새로운 자원을 반환합니다.

### 2️⃣ 서버가 요청받은 자원이 변경되지 않은 경우

![20260319_015935.jpg](20260319_015935.jpg)

- 서버는 메시지 본문 없이 상태 코드 304(Not Modified)를 통해 클라이언트에게 ‘자원이 변경되지 않았음을 알린다.
- Last-Modified 헤더로 함께 자원의 ‘마지막 변경 시점’을 알릴수있다.

### 3️⃣ 서버가 요청받은 자원이 삭제된 경우

![20260319_015951.jpg](20260319_015951.jpg)

- 서버는 상태 코드 404(Not Found)를 통해 요청한 ‘자원이 존재하지 않음’을 알린다.

## 원본 데이터 변경 여부(엔티티 태그) = Etag

- ‘자원의 버전’을 식별하기 위한 정보
- 자원이 변경될 때마다 Etag값이 변경 (반대면 변경X)

### If-None-Match

- 특정 자원의 엔티티 값이 변경되었는지를 묻는 헤더더
- 대표적인 HTTP 요청 헤더

### 상황

### 1️⃣ 서버가 요청받은 자원이 변경된 경우

- 서버는 상태 코드 200(OK)과 함께 새로운 자원을 반환한다.

### 2️⃣ 서버가 요청받은 자원이 변경되지 않은 경우

- 서버는 메시지 본문 없이 상태 코드 304(Not Modified)를 통해 클라이언트에게 ‘자원이 변경되지 않았음’을 알린다.

### 3️⃣ 서버가 요청받은 자원이 삭제된 경우

- 서버는 상태 코드 404(Not Found)를 통해 요청한 ‘자원이 존재하지 않음’을 알린다.

# 콘텐츠 협상

- 같은 자원에 대해 할 수 있는 여러 표현 중 클라이언트가 가장 적합한 자원의 표현을 제공하는 기술

자원 ) 서버와 클라이언트가 http메시지를 통해 주고받는 것

서버와 클라이언트가 http 메시지를 통해 주고받은 것 = 자원의 표현

표현 : 송수신 가능한 자원의 형태

### 인증서 ( 공개 키 인증서 )

- ‘당신이 통신을 주고받는 상대방은 틀림없이 당신이 의도한 대상이 맞다’ 라는 사실을 입증하기 위한 정보

### 인증 기관(CA)

- 인증서를 발급하고 검증, 저장 등 역할을 수행하는 공인 기관
- Certificate 메시지 & CertificateVerify 메시지
    - Certificate
        - 인증서 내용들이 포함된 메시지
    - CertificateVerify
        - 인증서의 내용이 올바른지 검증하기 위한 메시지

![20260319_121801.jpg](20260319_121801.jpg)

| 구분 | Certificate | CertificateVerify |
| --- | --- | --- |
| 의미 | 신분증 | 신분증 주인 인증 |
| 역할 | “나는 이런 서버야” | “이 인증서 진짜 내 거야” |
| 포함 | 공개키, 도메인 | 서명 |
| 타이밍 | 인증서 전달 단계 | 그 이후 검증 단계 |
| 한 줄 정리  | 신원 정보 | 그 신원을 실제로 증명하는 서명 |
- Finished 메시지

# 🔥 1️⃣ 쿠키 보완 (⭐ 거의 무조건 꼬리 질문)

### ✔ 쿠키 vs 세션 (필수)

| 구분 | 쿠키 | 세션 |
| --- | --- | --- |
| 저장 위치 | 클라이언트 | 서버 |
| 보안 | 취약 | 상대적으로 안전 |
| 용량 | 작음 | 비교적 큼 |
| 사용 | 자동 전송 | 서버에서 관리 |

👉 한 줄

> "쿠키는 클라이언트에 저장되고, 세션은 서버에 저장됩니다."
> 

---

### ✔ 쿠키 주요 옵션 (⭐ 중요)

- **HttpOnly**
    - JS 접근 불가 → XSS 방어
- **Secure**
    - HTTPS에서만 전송
- **SameSite**
    - CSRF 방어

👉 한 줄

> "쿠키는 HttpOnly, Secure, SameSite로 보안을 강화합니다."
> 

---

# 🔥 2️⃣ 캐시 보완 (⭐ 핵심)

## ✔ Cache-Control (중요)

- `max-age` → 캐시 유지 시간
- `no-cache` → 검증 후 사용
- `no-store` → 아예 저장 X

👉 한 줄

> "Cache-Control 헤더로 캐시 정책을 제어합니다."
> 

---

## ✔ 강한 캐시 vs 검증 캐시

👉 면접에서 좋아하는 포인트

- **강한 캐시**
    - 유효기간 안 지나면 서버 안 감
- **검증 캐시**
    - 서버에 확인 (If-Modified-Since, ETag)

👉 한 줄

> "강한 캐시는 바로 사용, 검증 캐시는 서버 확인 후 사용합니다."
> 

---

# 🔥 3️⃣ 콘텐츠 협상 보완

## ✔ 서버 주도 vs 클라이언트 주도

- 클라이언트 주도: Accept 헤더
- 서버 주도: 서버가 결정

👉 한 줄

> "콘텐츠 협상은 클라이언트 요청 기반으로 서버가 최적 표현을 선택합니다."
> 

---

# 🔥 4️⃣ HTTPS / TLS 보완 (⭐ 핵심 질문 많이 나옴)

## ✔ 대칭키 vs 비대칭키 (⭐ 필수)

👉 면접 단골

- 비대칭키 → 처음 키 교환
- 대칭키 → 실제 데이터 암호화

👉 한 줄

> "TLS는 비대칭키로 키를 교환하고, 이후 대칭키로 통신합니다."
> 

---

## ✔ 왜 둘 다 쓰냐?

👉 핵심 답변

- 비대칭 → 안전하지만 느림
- 대칭 → 빠름

👉 한 줄

> "보안성과 성능을 동시에 확보하기 위해 둘 다 사용합니다."
> 

---

## ✔ HTTPS 장점 3가지 (외우면 좋음)

- 기밀성 (암호화)
- 무결성 (위변조 방지)
- 인증 (서버 신뢰)

👉 면접 한 줄

> "HTTPS는 기밀성, 무결성, 인증을 제공합니다."
>

# 🔥 6️⃣ 보안 공격 관련 (⭐ 차별화 포인트)

## ✔ XSS / CSRF

👉 쿠키랑 같이 물어봄

- **XSS**
    - 스크립트 공격 → 쿠키 탈취
- **CSRF**
    - 사용자가 의도하지 않은 요청

👉 한 줄

> "XSS는 스크립트 공격, CSRF는 사용자 위조 요청입니다."
> 

---

# 🔥 7️⃣ 면접에서 “와 이거까지 아네?” 포인트

## ✔ 세션 vs JWT (조금 고급)

- 세션 → 서버 저장
- JWT → 클라이언트 저장

👉 한 줄

> "JWT는 서버 상태 없이 인증을 처리할 수 있습니다."
