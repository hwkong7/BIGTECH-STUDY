LAN을 넘어 다른 네트워크와 통신을 주고 받으려면, 네트워크 계층 이상의 기술이 필요!

# IP 목적과 특징

## 주소 지정과 단편화

**✅ 주소 지정**

> 네트워크 간의 통신 과정에서 호스트를 특정하는 것
> 

![image.png](attachment:3a08071e-1d90-4420-a361-2a3657f33502:image.png)

- IP를 통해 이루어짐 → 패킷 헤더 확인
- 송신지 IP, 수신지 IP 주소
    - 송수신자를 식별할 수 있는 IP 주소가 명시
    - 4바이트 → 0~255 범위의 10진수 4개
    - 점으로 구분되는 것을 옥텟이라고 표현
- IP : 수신 주소, 발신 주소
- MAC : 수신인, 발신인
- 라우터
    - 서로 다른 네트워크가 속한 두 호스트가 통신
    - IP 주소를 바탕으로 목적지까지 IP 패킷을 전달하는 장치
    - 라우팅 → IP 패킷을 전달할 최적의 경로를 결정하고 해당 경로로 패킷을 보냄

**✅ 단편화**

> 데이터를 여러 IP 패킷으로 올바르게 쪼개어 보내는 것
> 
- MTU
    - 최대 전송 단위
    - MTU보다 클 경우 패킷을 여러 패킷으로 쪼개서 전송 → 수신지에서 재조합
    - 일반적으로 1500바이트
        
        ![image.png](attachment:d71ea168-fa9b-47d5-8248-72b75883d869:image.png)
        
- 식별자
    - 특정 패킷이 어떤 데이터에서 쪼개진 패킷인지 식별
    - 같은 정보에서 쪼개진 패킷 → 같은 식별자 공유
- 플래그
    - 3비트로 구성
    - 1번째 비트
        - 항상 0으로 안씀
    - 2번째 비트
        - DF
            - IP 단편화를 수행하지 말라
            - 네트워크 MTU보다 크면 폐기 + ICMP 오류 발생
    - 3번째 비트
        - MF
        - 단편화된 패킷이 더 있다
        - 마지막 패킷만 0
- 단편화 오프셋
    - 특정 패킷이 초기 데이터에서 얼마나 떨어져 있는지가 명시된 필드
    - 단편화되어 전송되는 패킷을 목적지에서 재조합하기 위해 패킷의 올바른 순서를 나타내는데 사용

## 신뢰할 수 없는 통신과 비연결형 통신

IP는 신뢰할 수 없는 프로토콜 + 비연결형 프로토콜

**✅ 신뢰할 수 없는 프로토콜**

> 패킷이 수신지까지 제대로 전송되었다고 보장하지 않는 프로토콜
> 
- 패킷이 유실되거나 순서대로 도착하지 않아도 조치 X

**✅ 비연결형 프로토콜**

> 패킷을 주고받기 전에 사전 연결 과정을 거치지 않는다
> 
- 상대 호스트의 수신 가능 여부는 고려하지 않고 수신자를 향해 패킷을 전송

# IP 주소의 구조

네트워크 계층은 LAN 간 통신을 가능하게 한다!

이는 IP 주소로 이어지는데, 

- 네트워크 주소 : 호스트가 속한 네트워크를 특정
- 호스트 주소 : 네트워크에 속한 호스트를 특정

네트워크 주소를 표현하는 크기와 호스트를 표현하는 크기가 유동적일 수 있음

## 클래스풀 주소 체계

**✅ 클래스**

> 네트워크의 크기에 따라 유형별로 IP 주소를 분류하는 기준
> 
- 어떤 클래스에 속한 IP 주소인지를 알면 네트워크,호스트 부분이 어느 정도 크기인지 알 수 있음
- 주로 쓰는 종류는 A,B,C가 있음

![image.png](attachment:94ce0ff8-00c2-4c58-9782-73585413aba3:image.png)

![image.png](attachment:469e9f3b-4b62-4281-9663-bdc2684c6107:image.png)

## 클래스리스 주소 체계와 서브넷 마스크

클래스별 네트워크 크기가 고정 → 고정된 크기 외에 다른 크기의 네트워크를 구성할 수 없음

- A클래스
    - 할당 가능한 주소 : 1,600만 개
- B클래스
    - 6만 개
- C클래스
    - 200개

> 만약 300명의 직원이 사용 → B클래스 사용 → 낭비!!

클래스를 사용하지 않고 네트워크와 호스트를 구분하는 방식
> 

![image.png](attachment:1ad0d96a-843f-4aad-8dfc-954b79d041b1:image.png)

**✅ 서브넷 마스크**

> 서브넷을 구분하는 비트열을 말함
> 
- 서브넷
    - 네트워크 주소로 구분할 수 있는 네트워크의 부분집합
- IP 주소상에서 네트워크 주소를 1로 , 호스트 주소를 0으로 표기

![image.png](attachment:b0ba9811-27d0-4d79-8ecb-da878f130da8:image.png)

- 서브넷 마스크와 IP 주소 간에 비트 AND 연산을 수행하면 IP 주소 내의 네트워크 주소를 알아낼 수 있음

# 공인 IP 주소와 사설 IP 주소

**✅ 공인 IP 주소**

> 전 세계에서 고유한 주소
> 
- ISP나 공인 IP 주소 할당 기관을 통해 할당받을 수 있음
- 구글, 네이버 등 서버와 패킷을 주고 받으려면 호스트가 속한 네트워크 밖에서 사용할 공인 IP

**✅ 사설 IP 주소**

![image.png](attachment:48abd395-ecfd-436f-a830-9ea247689e68:image.png)

> 사설 네트워크에서 사용하기 위한 IP 주소
> 
- 외부 네트워크에 공개되지 않은 네트워크
- 라우터를 통해 할당받아 공유기를 통해 구성된 LAN 대부분은 사설 네트워크에 해당
- 사설 IP로 사용하도록 특별히 예약된 IP 주소 공간이 있음

# IP 주소의 할당

## 정적 할당

> 직접 수작업으로 IP 주소를 부여하는 방식
> 

![image.png](attachment:d818c4a3-125e-4599-9909-0640d25782ad:image.png)

- IP 주소, 서브넷 마스크, 게이트웨이 주소, DNS 주소가 필요
- 게이트웨이
    - 서로 다른 네트워크를 연결하는 하드웨어/소프트웨어 수단
    - 기본 게이트웨이 : 호스트가 속한 네트워크의 외부로 나가기 위한 첫 기본 경로
    - IP 할당 맥락에서 사용되는 게이트웨이 용어 → 기본 게이트웨이 → 공유기의 주소

## 동적 할당

> 프로토콜을 통해 자동으로 IP 주소를 부여하는 방식
> 
- DHCP 프로토콜을 사용
- DHCP 서버와 메세지를 주고받으며 동적 IP 주소를 할당받음
    - 라우터가 역할을 수행
- 사용가능한 기간이 정해져있음 → 기간 지나면 회수
- 할당 받을 때마다 다른 주소를 받을 수 있음
- 기간이 끝나기 전에 연장 가능

# IP 전송 특징의 보완 : ICMP

> IP 패킷의 전송 과정에 대한 피드백 메세지를 얻기 위해 사용하는 프로토콜
> 

![image.png](attachment:c6f053f0-127a-457e-bac3-3728be37bbbf:image.png)

- 전송 과정에서 발생한 오류 보고
    - 네트워크 도달 불가 :
        - 네트워크 장비가 패킷을 받았는데 해당 패킷을 어떤 네트워크로 전송해야할지 알 수 없음
    - 단편화가 필요하지만 DF가 1로 설정되어 단편화할 수 없음
        - 처리하기 너무 큰 패킷을 받았는데 DF 플래그가 설정되어 있어 단편화가 불가능할 경우
    - 시간 초과
        - TTL 필드가 0이 되어 도달 불가능
- 네트워크에 대한 진단 정보

# IP 주소와 MAC 주소의 대응:ARP

> 동일 네트워크 내에 있는 송수신 대상의 IP 주소를 통해 MAC 주소를 알아내는 프로토콜
> 

## ARP 요청 메세지

- 브로드캐스트
- 알고 싶은 MAC 주소에 대응되는 IP 주소가 포함
- 자신과 관련 없는 IP 주소일 경우 무시
- 자신의 IP 주소일 경우 응답 메세지 전송

## ARP 응답 메세지

- 호스트의 MAC 주소가 포함

## ARP 테이블

- IP 주소, MAC 주소 쌍을 기억
