# 쿠키

> 서버에서 생성되어 클라이언트 측에 저장되는 이름,값 쌍 형태의 데이터

Stateless 특성을 보완하기 위한 대표적인 수단
> 
- 서버는 쿠키를 생성하여 클라이언트에 전송
- 클라이언트는 쿠키를 저장해 두었다가 추후 같은 서버에 요청을 보낼 때 요청 메세지에 쿠키를 포함하여 전송

✅ 응답 메세지

- 이름, 값, 속성이 명시
- Set-Cookie 헤더가 사용
- 여러 개의 헤더가 사용됨

✅ 요청 메세지

- 특정 서버로부터 쿠키를 전달 받음 → 해당 서버에 요청을 보낼 때 전달받은 쿠키를 자동으로 전송

✅ 속성 값

- 쿠키는 기본적으로 이름,값 쌍으로 구성되어 있지만, 속성값도 포함
- 도메인 경로
    - domain, path
- 유효기간
    - Expires
        - 명시된 유효기간, 날짜 형식
    - Max-Age
        - 초 단위 유효기간
- 보안
    - Secure
        - HTTPS를 통해서만 쿠키를 송수신하도록 하는 속성
    - HttpOnly
        - 자바스크립트를 통한 쿠키의 접근을 제한
        - 오직 HTTP 송수신을 통해서만 쿠키에 접근하도록 하는 방식

# 캐시

> 메모리에 접근하는 시간을 줄이기 위해 자주 참조되는 내용을 저장하는 장치
> 
- 이미지를 임시 저장하면 재활용하여 더 빠르게 자원에 접근
- 캐시된 데이터는 유효 기간이 설정되어 있다
    - Expires
    - Max-Age
- 유효 기간이 부여되는 이유
    - 캐시를 참조하는 사이 서버의 원본 데이터가 변경되어 일관성이 깨질 수 있음
    - 유효 기간을 설정하고 재요청함으로써 캐시 신선도를 검사할 수 있음
- 유효 기간이 만료되면 서버에 원본 자원이 변경된 적이 있는지 질의
    - 유효 기간이 만료되도 여전히 최신 데이터면 같은 자원을 받을 필요가 없음

✅ If-Modified-Since

<aside>

이 시점 이후로 자원에 변경이 있으면 그때만 변경된 자원을 메세지 본문으로 응답

1️⃣ 서버가 요청받은 자원이 변경된 경우

- 상태 코드 200과 함께 새로운 자원을 반환

2️⃣ 서버가 요청받은 자원이 변경되지 않은 경우

- 메세지 본문 없이 304를 통해 자원이 변경되지 않았음을 알림
- Last-Modified 헤더로 자원의 마지막 변경 시점을 알릴 수 있음
- 캐시 데이터 그대로 사용 가능

3️⃣ 서버가 요청받은 자원이 삭제된 경우

- 404를 통해 요청한 자원이 존재하지 않음을 알림
</aside>

- 200과 함께 새로운 자원 반환

2️⃣ 서버가 요청받은 자원이 변경되지 않은 경우

- 304 리턴 후 캐시 사용

# 콘텐츠 협상

> 같은 자원에 대해 할 수 있는 여러 표현 중 클라이언트가 가장 적합한 자원의 표현을 제공하는 기술

동일한 자원을 요청해도 한국어, 영어로 된 표현이 응답될 수 있음
> 
- Accept
    - 선호하는 미디어 타입을 나타내는 헤더
- Accept-Language
    - 선호하는 언어를 나타내는 헤더
- Accept-Encoding
    - 선호하는 인코딩 방식을 나타내는 헤더
- 클라이언트가 우선순위를 반영하여 여러 표현에 대한 선호도를 서버에 알릴 수 있음
    - q 값으로 표시

# HTTPS

> HTTP에 SSL 혹은 TLS 프로토콜 동작을 추가하여 암호화 통신이 이루어짐
> 

![image.png](attachment:dfe7ec93-2a8b-44a9-a316-edf774355869:image.png)

1️⃣ TCP 쓰리 웨이 핸드셰이크

**2️⃣ TLS 핸드셰이크**

- 암호화 통신을 위한 키를 생성/교환할 수 있음
- 인증서 송수신과 검증이 이루어질 수 있음

3️⃣ 메세지 송수신

**✅ 암호화 통신을 위한 키**

- 평문을 암호화하거나 암호문을 복호화하려면 키라는 정보가 필요
- 암호화 통신을 수행하는 두 호스트만 알고 있어야 하는 정보
- TLS 핸드셰이크 과정에서 메세지를 주고받으며 생성,교환

- ClientHello
    - 암호화된 통신을 위해 서로 맞춰야 할 정보
    - 지원하는 TLS 버전, 사용 가능한 암호화 알고리즘과 해시 함수, 클라이언트 난수
    - 암호 스위트
- ServerHello
    - 제시된 정보들을 선택하는 메세지
    - TLS 버전, 암호 스위트 정보, 난수
    - 결정된 정보를 통해 암호화에 사용할 키를 만들어 암호화에 사용

**✅ 인증서의 송수신과 검증**

> 당신이 통신을 주고받는 상대방은 틀림없이 당신이 의도한 대상이 맞다→ 입증하기 위한 정보
> 
- www.hanbit.co.kr와 메세지 주고받을 때 → 대상이 틀림없다!
- 제3의 기관이 보장 → CA

Certificate

- 인증서 서명 값 등 앞서 제시한 인증서 내용

CertificateVerify

- 인증서 내용이 올바른지 검증하기 위한 메세지

이렇게 서버와 클라이언트는

1. 암호화에 사용할 키 획득
2. 서로가 틀림없다는 사실을 인증
3. TLS 핸드셰이크의 마지막인 Finished 메세지를 주고 받음
4. 얻어낸 키를 기반으로 암호화된 데이터를 주고받음
